# SicoobSso

Reusable client side of Sicoob's in-house Single Sign-On. The identity provider
("Tools") authenticates users; this gem authenticates a host Rails app against it.

As of v0.3.0 this is a **Rails Engine**: it ships the sessions controller and views.
The host app provides `Session`, `Current`, and a user provisioner; the gem wires up
the routes and handles the login flow.

The gem supports two login strategies, selected per host:

- **`:redirect`** (default) — classic OAuth-style redirect. The host sends the user to
  the IdP (`GET /sso/authorize`), the user authenticates there, and the IdP redirects
  back with a code the host exchanges (`POST /sso/token`).
- **`:push_approval`** — passwordless on the host. The user enters only their e-mail;
  the IdP shows an approval request in its in-app notification center; once the user
  approves, the host exchanges the result for claims. Use this when the user can reach
  the IdP to approve (e.g. internal web apps).

Switching strategies is a single config change — no controller or view scaffolding
needed in the host app.

## Installation

```ruby
gem "sicoob_sso", git: "https://github.com/sicoobmetropolitano/sicoob_sso.git"
```

## What's in the box

| Component | Type | Responsibility |
|-----------|------|----------------|
| `SicoobSso::Configuration` / `SicoobSso.configure` | config | Provider URL, client credentials, redirect URI, user provisioner, login path, **auth strategy** |
| `SicoobSso::IdentityProvider` | module function | `authorize_url(state:)`, `exchange_code(code)` (redirect); `create_auth_request(email:)`, `poll_auth_request(request_id:)` (push-approval) |
| `SicoobSso::Authentication` | controller concern | `current_user`, `user_signed_in?`, `authenticate_user!`, `sign_in`, `sign_out`, `resume_session` (token cookie) |
| `SicoobSso::SessionsControllerConcern` | controller concern | login actions shared by all strategies |
| `SicoobSso::SessionsController` | engine controller | engine-provided sessions controller (host can override with its own) |
| `app/views/sicoob_sso/sessions/` | engine views | `new.html.erb` (e-mail form) and `waiting.html.erb` (polling page) — overridable by the host |

## Integration checklist (new client)

1. Add the gem and `bundle install`.
2. Provide the host `Session` model and `Current` (see "Host requirements").
3. Add `config/initializers/sicoob_sso.rb` with at least the `provisioner` (and
   `auth_strategy` if push-approval).
4. Call `sicoob_sso_routes` in `config/routes.rb`.
5. Add `before_action :authenticate_user!` to `ApplicationController` (the auth concern
   is auto-included by the engine — no manual `include` needed).
6. Register the app on the Tools IdP per environment (`rake sso:register`) and set
   `SSO_CLIENT_SECRET` from its output.
7. Set the other env vars (`SSO_PROVIDER_URL`, `SSO_CLIENT_ID`, `SSO_REDIRECT_URI`,
   `SSO_AUTH_STRATEGY`) per environment.

Each step is detailed below.

## Configuration

```ruby
# config/initializers/sicoob_sso.rb
SicoobSso.configure do |c|
  c.provider_url   = ENV.fetch("SSO_PROVIDER_URL")   # e.g. https://tools.example
  c.client_id      = ENV.fetch("SSO_CLIENT_ID")
  c.client_secret  = ENV.fetch("SSO_CLIENT_SECRET")
  c.redirect_uri   = ENV.fetch("SSO_REDIRECT_URI")   # redirect strategy only
  c.login_path     = "/login"                        # string, or a callable resolved in the controller
  c.auth_strategy  = :push_approval                  # :redirect (default) | :push_approval
  c.provisioner    = ->(claims) { User.provision_from_sso(claims) }
end
```

Every setting falls back to an ENV default
(`SSO_PROVIDER_URL`, `SSO_CLIENT_ID`, `SSO_CLIENT_SECRET`, `SSO_REDIRECT_URI`,
`SSO_AUTH_STRATEGY`), so the initializer only needs to set what differs (typically the
`provisioner` and, for push-approval hosts, `auth_strategy`).

`login_path` may be a string or a callable. A callable is evaluated in the controller
instance, so route helpers work: `c.login_path = -> { login_path }`.

## Environment variables

| Var | What it is | Where it comes from |
|-----|-----------|---------------------|
| `SSO_PROVIDER_URL` | Base URL of the Tools IdP | The Tools deployment URL for that environment |
| `SSO_CLIENT_ID` | This app's identifier at the IdP | Chosen at registration (e.g. `onask`) |
| `SSO_CLIENT_SECRET` | This app's password at the IdP | **Generated by the IdP at registration** (see below). Store as a secret — never in git |
| `SSO_REDIRECT_URI` | Callback URL (redirect strategy) | `https://<this app>/sso/callback` — must byte-match what was registered |
| `SSO_AUTH_STRATEGY` | `redirect` (default) or `push_approval` | Per host |
| `SSO_APP_NAME` | App title shown on the default login/waiting views | Optional; falls back to "Entrar" |

Set these in your secret store / deploy config per environment. Only `SSO_CLIENT_SECRET`
is sensitive; the rest can live in plain config.

## Registering the client at the IdP (Tools)

A host can't authenticate until it's registered as a client on Tools. Registration
happens **on the Tools app**, once **per environment** (each environment has its own
database, so each produces its own secret):

```bash
# run on the Tools host (the IdP) for the target environment
rake "sso:register[<name>, <client_id>, https://<this app>/sso/callback]"
```

It prints, once (not recoverable):

```
client_id=<client_id>
client_secret=<generated>
```

Put that `client_secret` into this app's `SSO_CLIENT_SECRET` for that environment. The
`redirect_uri` passed to `sso:register` must exactly match this app's `SSO_REDIRECT_URI`.

**Both strategies need registration.** `:push_approval` has no browser redirect, but the
host still authenticates to the IdP with `client_id` + `client_secret` on the back channel,
so the client row (and its secret) is required either way; pass this app's callback as the
`redirect_uri`.

## Host requirements (both strategies)

The gem references two host constants, resolved where the concerns are included.

### `Session` model

```ruby
class Session < ApplicationRecord
  belongs_to :user
  has_secure_token :token

  def active?         = # your freshness rule, e.g. last_active_at > 30.days.ago
  def touch_activity! = update_column(:last_active_at, Time.current)
end
```

Required surface: `Session.find_by(token:)`, instance `token`, `active?`,
`touch_activity!`, and `user.sessions.create!(user_agent:, ip_address:)`.

### `Current` (ActiveSupport::CurrentAttributes)

```ruby
class Current < ActiveSupport::CurrentAttributes
  attribute :session, :user
end
```

### A user provisioner

A callable that takes the IdP's `"user"` claims hash and returns a host `User`
(find-or-create). Wired through `SicoobSso.config.provisioner`.

### Base controller

`SicoobSso::Authentication` is auto-included into `ActionController::Base` by the engine
(like Devise's controller helpers), so you do **not** write the `include`. You only choose
which controllers require login with `before_action :authenticate_user!`:

```ruby
class ApplicationController < ActionController::Base
  before_action :authenticate_user!
end
```

(If you need it elsewhere — e.g. an `ActionController::API` base — include
`SicoobSso::Authentication` manually there.)

## Routes

Add a single helper call to `config/routes.rb`:

```ruby
Rails.application.routes.draw do
  sicoob_sso_routes
  # ... your other routes
end
```

This mounts all six SSO routes regardless of the active strategy:

| Verb | Path | Action | Helper |
|------|------|--------|--------|
| GET | `/login` | `sicoob_sso/sessions#new` | `login_path` |
| POST | `/login` | `sicoob_sso/sessions#create` | — |
| GET | `/sso/waiting` | `sicoob_sso/sessions#waiting` | `sso_waiting_path` |
| GET | `/sso/poll` | `sicoob_sso/sessions#poll` | `sso_poll_path` |
| GET | `/sso/callback` | `sicoob_sso/sessions#callback` | — |
| DELETE | `/logout` | `sicoob_sso/sessions#destroy` | `logout_path` |

For `:redirect`, the `new` action skips rendering and redirects immediately to the IdP;
`/login` (POST), `/sso/waiting`, and `/sso/poll` are unused. For `:push_approval`,
`/sso/callback` is unused (the IdP calls back to Tools, not to this app).

## Views (overridable)

The engine ships default views for the push-approval flow:

- `app/views/sicoob_sso/sessions/new.html.erb` — a minimal e-mail-only form
- `app/views/sicoob_sso/sessions/waiting.html.erb` — a polling page with inline JS

To override either, create the file at the same path inside the host app. Rails resolves
host views before engine views.

For `:redirect`, `new` redirects to the IdP immediately — no view is rendered.

## Flow: `:redirect`

1. `GET /login` → `new` generates a state nonce, stores it in the session, redirects to the IdP.
2. IdP redirects to `GET /sso/callback` → `callback` verifies state, exchanges code for
   claims, runs provisioner, signs in, redirects to `return_to` (or `/`).
3. `DELETE /logout` → `destroy` signs out.

## Flow: `:push_approval`

1. `GET /login` → `new` renders the e-mail form.
2. `POST /login` → `create` calls `IdentityProvider.create_auth_request(email:)`, stores
   the `request_id`, redirects to `GET /sso/waiting`.
3. `waiting` renders a page that polls `GET /sso/poll` every 2 seconds.
4. `poll` (JSON) calls `IdentityProvider.poll_auth_request(request_id:)`:
   - While pending: `{ status: "pending" }`
   - On approval: exchanges code, signs in, returns `{ status: "approved", redirect_to: ... }`
   - Denied/expired/error: returns `{ status: "denied"|"expired"|"error" }` — the client
     navigates back to `/login`.

The IdP side (request creation, approval notification, approve/deny, signed single-use
code) lives in the Tools app.

## Errors

- `SicoobSso::Error` — base error
- `SicoobSso::ExchangeError` — raised by `IdentityProvider` on a non-2xx response from
  the IdP (token exchange, auth-request creation, or polling)

## Security

- **Back channel uses TLS + timeouts.** All IdP calls (`/sso/token`,
  `/sso/auth_requests`, polling) go over `https` with certificate verification
  (`VERIFY_PEER`) and bounded open/read timeouts (5s / 10s), so a hung or
  unreachable IdP can't block app threads. Always set `SSO_PROVIDER_URL` to an
  `https` URL in production.
- **Rate-limit the login endpoint.** The gem doesn't throttle requests; the host
  owns that. The host controller that includes `SicoobSso::SessionsControllerConcern`
  should rate-limit `POST /login` (push-approval) so the IdP isn't flooded with
  auth requests:

  ```ruby
  class Sso::SessionsController < ApplicationController
    include SicoobSso::SessionsControllerConcern

    rate_limit to: 10, within: 1.minute, only: :create
  end
  ```

  (`rate_limit` is built into Rails 8's `ActionController::Base`.)
- **Default views are provided** (`new`, `waiting`) — plug-and-play out of the
  box and overridable: drop a file at the same path in the host app to replace
  either one.

## Development

```bash
bundle install
bundle exec rake test
```

`IdentityProvider` and `Configuration` are unit-tested here. The controller concerns are
integration-tested in the host application (they need a Rails controller context,
`Session`, and `Current`); the gem suite only verifies they load and expose the expected
methods.
